## Sesja 2026-05-24

## Pomysł

Gra pinball generowana z layoutu strony www. Gracz wpisuje URL, silnik analizuje DOM i tworzy poziom — kulka odbija się od elementów strony (headery, karty, obrazki, przyciski). Footer to podłoga z flipperami.

**Docelowy model biznesowy:** płatny widget B2B — firma embedduje na swojej stronie jako interaktywny gadżet / Easter egg. Z możliwością personalizacji powiązanej z treścią strony (branded bumpery, efekty trafień w CTA, logo jako specjalny bumper).

---

## Decyzje techniczne

### Architektura widgetu (client-side)

Widget działa w całości w przeglądarce odwiedzającego — zero crawlowania, zero problemów z Cloudflare.

```html
<script src="https://cdn.twojadomena.com/widget.js" data-key="KLUCZ_KLIENTA"></script>
```

Flow:
1. Widget ładuje config klienta z CDN (JSON po kluczu API)
2. Czyta DOM strony → `getBoundingClientRect()` na elementach
3. Tworzy `<canvas>` overlay (fixed, fullscreen, z-index wysoki)
4. Matter.js tworzy świat fizyki: prostokąty z DOM = statyczne ciała, kulka = dynamiczne ciało
5. Gra działa lokalnie — zero ruchu sieciowego podczas grania

Scroll: przy aktywacji gry blokujemy scroll, pracujemy na elementach widocznych w momencie uruchomienia.

### Demo "wpisz URL" (narzędzie sprzedażowe)

Osobna strona Next.js:
- Backend: Playwright → screenshot + JSON z prostokątami elementów
- Frontend: screenshot jako tło + Matter.js canvas overlay
- Cloudflare: problem dla ~30-40% stron, ale dla demo wystarczy

### Stack

| Warstwa | Technologia |
|---|---|
| Backend | Next.js + Playwright |
| Fizyka | Matter.js (~50KB) |
| Renderowanie | Canvas API |
| Stylowanie | Tailwind 4 |

Żadnych nowych technologii poza Matter.js.

### Personalizacja (B2B)

Dashboard klienta:
- Klucz API → config JSON
- Kolory, skin kulki
- Specjalne bumpery (kształt logo)
- Efekty trafień: trafienie w CTA = bonus punktowy + animacja
- Które selektory CSS = przeszkody

## Sesja 2026-05-28

### Co zrobione

**Cel/meta gry — zmiana z "najdalszej komórki" na "dziurę w prawej ściance":**

Michał pokazał screenshoty z zaznaczonymi pozycjami:
- **Czerwone kółko** = poprawna pozycja startowa (lewy górny róg) — już była ok, nie zmieniano
- **Żółte kółko** = meta — geometryczna przerwa w prawej ściance labiryntu (dolna prawa strona)

**Zmiany w `maze.html`:**

1. **Dodano `<canvas id="dbg-canvas">`** — overlay 2D na WebGL canvas
2. **Zmodyfikowano RIM exclusion w `buildGrid()`** — usunięto wykluczenie prawej krawędzi (`|| xi >= GCOLS - RIM`), żeby naturalne przerwy w ściance (dziury) były passable
3. **Auto-detekcja dziury** — po RIM loopie skanujemy prawą strefę (xi >= GCOLS-RIM), szukamy passable cells; środek ciężkości = holeX, holeZ; logowane do konsoli
4. **`finishSetup()` — nowa logika celu:**
   - `MANUAL_START` / `MANUAL_GOAL` = overrides ręczne (stałe w constants)
   - `holeFound` = True → goal = dziura
   - fallback = stara logika Manhattan
5. **Warunek wygranej:** `ball.x > holeX - 0.35 && |ball.z - holeZ| < 0.9` (kulka wtoczyła się w otwór)
6. **`drawDebugGrid()`** — rysuje passable cells (zielone), ściany (czerwone), dziurę (pomarańczowy kwadrat + label z coords) — aktywna gdy `SHOW_DEBUG_GRID = true`

### Stan na koniec sesji

- Zmiany zaimplementowane, **niezweryfikowane wizualnie** — Michał ma sprawdzić overlay i potwierdzić czy dziura jest wykryta we właściwym miejscu
- `SHOW_DEBUG_GRID = true` — do wyłączenia przed shipowaniem
- `maze.html` nadal niezdeployowane na VPS

### Otwarte zadania

1. Weryfikacja: czy dziura auto-detekuje się w miejscu żółtego kółka?
